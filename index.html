1. Introducción a la Elicitación de Requisitos

¿Qué es la Elicitación de Requisitos?

La elicitación de requisitos es el proceso de descubrir, capturar y entender las necesidades y restricciones de los interesados (stakeholders) para un sistema o proyecto. Es un paso fundamental en el desarrollo de software, ya que un buen entendimiento de los requisitos es la base para construir un producto exitoso.

Imagina que estás construyendo una casa. Si no entiendes lo que el cliente quiere (cuántas habitaciones, qué estilo, qué presupuesto), es muy probable que construyas algo que no le sirva. Lo mismo ocurre con el software: sin requisitos claros, el producto final no cumplirá las expectativas.

¿Por qué Técnicas Centradas en la Documentación?

Las técnicas de elicitación de requisitos se clasifican de diversas formas. En esta guía, nos enfocaremos en aquellas centradas en la documentación. Esto implica que la información se recopila principalmente a través de textos, formularios, o la observación sistemática que luego se registra detalladamente. Son técnicas muy útiles cuando:

  • Se necesita recopilar información de un gran número de personas.
  • Los interesados están geográficamente dispersos.
  • Se necesita una forma estandarizada y estructurada de recopilar datos.
  • El analista no puede interactuar directamente con todos los interesados o necesita observar el comportamiento real sin intervención.

2. Cuestionario

Explicación Teórica

Un cuestionario es un conjunto de preguntas escritas que se utilizan para recopilar información específica de un grupo de personas. Se aplica a individuos o pequeños grupos con los que se busca una interacción más personal o detallada, o cuando se necesita información muy particular de cada respondente. Aunque puede parecer similar a una encuesta, el cuestionario a menudo busca una comprensión más profunda de un tema específico con menos participantes.

  • Cuándo usar: Para obtener información detallada de un número limitado de expertos o usuarios clave; para confirmar suposiciones; cuando el tiempo para entrevistas es limitado.
  • Ventajas: Permite obtener información detallada, flexibilidad en el tipo de preguntas, relativamente bajo costo de implementación.
  • Desventajas: Requiere tiempo para analizar respuestas abiertas, puede ser subjetivo, no apto para grandes muestras.
  • Tipos de Preguntas:
    • Abiertas: Permiten al encuestado expresarse libremente con sus propias palabras (ej: "¿Qué mejoras sugeriría?"). Son útiles para obtener perspectivas cualitativas y no esperadas.
    • Cerradas: Restringen las respuestas a opciones predefinidas (ej: "Sí/No", escala Likert, selección múltiple). Facilitan el análisis cuantitativo y la comparación.

3. Encuesta

Explicación Teórica

Una encuesta es un método de recopilación de datos diseñado para obtener información de un gran número de personas (una muestra representativa de una población). A diferencia del cuestionario, que puede ser más profundo y cualitativo en un grupo pequeño, la encuesta busca datos cuantitativos para identificar patrones, tendencias y opiniones a gran escala.

  • Cuándo usar: Para recopilar datos estandarizados de una población grande y diversa; para medir opiniones o preferencias generales; para validar hallazgos de otras técnicas.
  • Ventajas: Permite obtener datos cuantitativos y estadísticos, eficiencia para grandes muestras, anonimato que puede fomentar respuestas honestas.
  • Desventajas: Menos profundidad en las respuestas, diseño cuidadoso para evitar sesgos, no permite aclaraciones en tiempo real.
  • Diseño: Las preguntas suelen ser predominantemente cerradas para facilitar la tabulación y el análisis estadístico, permitiendo la agregación de respuestas.

4. Técnicas de Observación

Explicación Teórica

Las técnicas de observación implican ver y registrar cómo los usuarios interactúan con un sistema existente (o un prototipo), o cómo realizan sus tareas diarias en su entorno de trabajo, sin que el analista intervenga activamente. Es una técnica valiosa para descubrir requisitos no expresados o para entender el contexto real de uso de un sistema, es decir, lo que los usuarios realmente hacen, no solo lo que dicen que hacen.

  • Cuándo usar: Cuando los usuarios tienen dificultades para articular sus necesidades; cuando el sistema existente no tiene documentación clara; para entender flujos de trabajo reales y el comportamiento natural; para identificar ineficiencias o errores de usabilidad.
  • Ventajas: Revela requisitos implícitos o no verbalizados; reduce la subjetividad de lo que el usuario dice; permite ver el contexto completo de uso; identifica problemas de usabilidad o cuellos de botella.
  • Desventajas: Consume mucho tiempo; el observador puede influir en el comportamiento (efecto Hawthorne); los usuarios pueden sentirse incómodos o ansiosos; no es escalable para muchos usuarios simultáneamente.
  • Tipos:
    • Pasiva/No Participativa: El observador no interviene en la actividad del usuario. Simplemente registra lo que ve y oye, sin interactuar.
    • Participativa: El observador se involucra en la actividad, quizás realizando tareas junto al usuario o pidiendo explicaciones sobre la marcha para entender mejor los desafíos y el proceso.
    • Etnográfica: Inmersión profunda del observador en el entorno del usuario por un periodo prolongado, buscando comprender la cultura, los valores y el contexto completo del trabajo, más allá de tareas específicas.

5. Conclusión

Resumen y Reflexiones Finales

En esta guía, hemos explorado tres técnicas fundamentales de elicitación de requisitos centradas en la documentación:

  • Cuestionario: Ideal para recopilar información detallada, cualitativa y cuantitativa, de un grupo más pequeño, a menudo con una mezcla de preguntas abiertas y cerradas. Permite profundizar en temas específicos.
  • Encuesta: Perfecta para obtener datos cuantitativos y tendencias de una población grande y diversa, utilizando principalmente preguntas cerradas para facilitar el análisis estadístico y la generalización de resultados.
  • Técnicas de Observación: Cruciales para descubrir requisitos implícitos y entender el comportamiento real del usuario en su entorno natural, revelando lo que los usuarios hacen en lugar de lo que dicen.

La elección de la técnica adecuada (o la combinación de ellas) depende de varios factores, como el tamaño de la población a consultar, el tipo de información necesaria (cualitativa vs. cuantitativa), los recursos disponibles (tiempo, presupuesto) y el contexto específico del proyecto. A menudo, las mejores estrategias de elicitación de requisitos combinan varias técnicas para obtener una visión completa y robusta de las necesidades del sistema, complementando las debilidades de una técnica con las fortalezas de otra.
